# LLM X-Ray

Interactive web tool: type a prompt, watch a small LLM (Qwen3-1.7B) process it in real time (attention patterns, per-layer logit-lens predictions, token probabilities), and export a shareable card. Pitch: "Type a prompt. Watch an AI think. Share the X-ray."

Full product/design rationale lives in `LLM-XRAY-Deep-Dive.md` — read it for the *why*; this file is the *how to work here*.

## Architecture

Two apps, streaming between them over WebSocket:

```
backend/   FastAPI + PyTorch + HuggingFace Transformers (Qwen3-1.7B, CPU); Python 3.11 venv at backend/.venv
frontend/  Next.js 16 (App Router, TS, src/) + Tailwind v4 + Framer Motion + shadcn/ui + D3.js
```

Flow: prompt → WS → tokenize → **greedy autoregressive generation** (prompt pass primes a KV cache, then one token per `advance()`), with hooks capturing per-layer hidden states (+ `outputs.attentions`) at each step → batched logit lens (all layers' last-position vectors through `final_norm` + `lm_head` in one matmul) + attention rollout → stream JSON messages back per generated token → React renders incrementally as the model writes.

The core is the **hook engine** (`backend/app/xray_engine.py`): registers PyTorch forward hooks on every decoder block (residual stream) and reads attention weights `(batch, heads, seq, seq)` from the forward pass's `output_attentions`. Architecture-specific module paths are resolved by `ModelAdapter` (`backend/app/model.py`), so the engine is **model-agnostic** — `MODEL_NAME` can swap between `Qwen/Qwen3-0.6B-Base` and `gpt2` with no engine changes. This is the whole product — keep it correct and fast.

## WebSocket protocol

Endpoint `/ws/xray`. Client sends `{"prompt": "...", "thinking": bool}` (or a bare string; thinking defaults on). Server chat-templates the prompt so the model **answers**, and streams a greedy generation per run, in order:
1. `{"type": "meta", "data": {"num_layers", "num_heads", "thinking", "model_label"}}`
2. `{"type": "tokens", "data": {"tokens", "token_ids"}}` — the **chat-templated** prompt tokens (incl. `<|im_start|>` etc.)
3. `{"type": "prompt_attention", "data": {"attention"}}` — `(layers, P, P)`, **mean over heads**
4. One `{"type": "step", "data": {"step", "token", "token_id", "prob", "entropy", "phase": "think"|"answer", "trajectory": [{layer_idx, top_predictions}], "attention_row": (layers, P+step) mean-over-heads}}` **per generated token** (EOS is not surfaced as a step)
5. `{"type": "done", "data": {"generated_text", "num_steps", "stop_reason"}}`

An out-of-band `{"type": "error", "data": {"message"}}` can replace the stream at any point. Attention is mean-over-heads (no per-head / no rollout on the wire — rollout is computed client-side on demand). When changing this shape, update both `backend/app/serializer.py` and the frontend `useXRay` hook + `xray-protocol.ts` together — they are tightly coupled.

## Conventions

- Serialize tensors with **orjson**, not stdlib `json` (10x faster on the numpy payloads). Always `.detach().cpu().numpy()` before serializing; never ship grad-tracking tensors.
- Inference runs inline via asyncio — no Celery/queue. Each forward pass (prompt or one generated token) is dispatched via `asyncio.to_thread` and streamed as it lands; per-token ~0.06s on CPU. Overall length/time is bounded in the engine (`MAX_NEW_TOKENS`/`GEN_TIME_BUDGET_S`), not by a single run timeout. Profile before adding complexity.
- Frontend: dark mode is the default (visualizations are designed to pop on dark backgrounds). D3 owns the SVG/canvas drawing; React owns state and layout. Don't let React re-render fight D3 transitions. D3 drawing belongs in client components (`"use client"`) inside effects/refs.
- shadcn/ui is initialized (`components.json`, `src/lib/utils.ts`, `src/components/ui/`). Add primitives on demand with `npx shadcn@latest add <name>`; don't hand-roll components shadcn already provides.
- Hero visual is the **logit-lens trajectory** (`LensTrajectory.tsx` — per-layer top-1 guess sharpening with depth) for the selected generated token. The **generation timeline** + **confidence curve** are the always-visible spine; candidates ("almost said X") and attention attribution (raw↔rollout) are supporting views. Instrument palette in `src/lib/viz.ts` — no rainbows.

## Commands

Backend (Python 3.11 venv already created at `backend/.venv`):
- Run dev server: `cd backend && .venv/bin/uvicorn app.main:app --reload` (default port 8000; pass `--port N` if 8000 is taken)
- Install deps: `cd backend && .venv/bin/pip install -r requirements.txt` — note `torch`/`transformers`/`numpy` are listed but only installed from M1 onward (large download); M0 installs just the web layer
- Health check: `curl http://127.0.0.1:8000/api/health`

Frontend (Next.js 16):
- Run dev server: `cd frontend && npm run dev` (http://localhost:3000)
- Build / lint: `npm run build` / `npm run lint`
- Add a shadcn primitive: `cd frontend && npx shadcn@latest add <name>`

## Scope guardrails

- Model is Qwen3-1.7B (2.03B, 28 layers, 16 query heads) on CPU, fed **chat-templated prompts** so it *answers* (with an optional `<think>…</think>` reasoning trace) — ~0.13s/token; a thinking run is ~15–25s. The engine is model-agnostic (`ModelAdapter` resolves module paths; chat template + `<think>` markers are resolved at runtime with raw-encode/no-thinking fallbacks; `NUM_LAYERS`/`MODEL_LABEL` are frontend *fallbacks*, live values come from `meta`), so swapping `MODEL_NAME` (e.g. to `Qwen/Qwen3-0.6B-Base` or `gpt2`) needs no other code changes — a non-chat model just falls back to raw continuation. Don't reach for GPUs without a reason tied to the roadmap.
- Multi-model *comparison* (running two models side by side), embeds, and the example gallery are v1.1+ — don't build them into the core path prematurely. (The model-agnostic engine keeps that door open, but a single model is the v1 path.)
